# 星露谷物语过夜系统与夜间结算实现规划 (StardewCraft Overnight System)

## 1. 当前开发进度评估
目前 `StardewCraft` 的时间系统由 `StardewTimeManager` 管理，状态如下：
- **时间流挂载**：依赖于 Minecraft 原版的 `dayTime` 进行同步（0点=早6点，20000=凌晨2点）。
- **跨天动作**：时间到达凌晨 2:00 时调用 `advanceDay()`，目前仅包含强制倒地扣体力的逻辑，并且依次触发植物生长、机器刷新、农场动物更新和天气切换。
- **缺失的核心环节**：
  - 没有“上床睡觉”的时间维度冻结机制，导致玩家面临跨天时的突发怪物威胁。
  - 没有**基于栈(Stack)的夜间结算UI流**（技能升级、职业选择、出货清算）。
  - 夜间数据结算（如解锁配方、图鉴收集记录、血量瞬时回满）与前端逻辑脱节。

## 2. 完整过夜与夜间结算流程全景 (架构设计)
根据原版星露谷的逻辑，完整过夜必须改造为一个**基于客户端/服务端交互的异步状态机**。
流程严格遵循以下顺序：

### 阶段一：触发与世界冻结 (Trigger & Freeze)
- 玩家确认睡觉（或凌晨 2:00 强制晕倒）。
- **服务端维度冻结**：令星露谷世界的时间停止流逝，并可能给全体范围内玩家挂上隐身/无敌或传送至安全坐标，避免 UI 交互期间产生伤害或饥饿。

### 阶段二：结算菜单入栈与数据盘点 (Stack Building & Calculations)
服务端盘点今日数据，并将界面发往客户端**排队显示**：
- **技能经验盘点**：统计五项技能（农、采、挖、钓、战）是否突破了等级阈值，生成 `LevelUp[N]` 升级清单。
- **出货数据盘点**：统计全图出货箱内物品，按类别归算金币。
- **入栈顺序决定展示顺序**：原版底层采用栈（Stack）。如果包含出货系统，先压入 `SaveGame`（如果有），再压入 `ShippingMenu`，最后逆序压入所有 `LevelUpMenu`。这意味着玩家会**先看到所有升级页面，最后看到出货页面**。

### 阶段三：客户端弹栈交互 (Client UI Interaction)
客户端接收到网络包后，开始拉起 UI：
1. **技能升级与解锁阶段 (`LevelUpMenu`)**：
   - 界面弹出瞬间：向玩家配方库写入基于该等级解锁的**烹饪/制造配方**。
   - 到达 5级/10级 时：展示职业分支（Profession）左右选项。
   - 玩家点击确认瞬间：
     - 若选择了 `战士/防御者` 或只是 `战斗技能升级`，立刻提升 `maxHealth`。
     - 强制执行玩家**满血点、满体力点复活**，以此豁免白天的所有惩罚（前提是没有透支熬夜逻辑）。
     - 将相关成就或条件邮件（例如钓鱼2级的鱼竿信）塞入明早信箱队列。
2. **出货结算阶段 (`ShippingMenu`)**：
   - 统计并展示全部收入，进入滚动动画状态。
   - 在加载阶段，将所有卖出的物品 ID 写入图鉴 (`player.shippedBasic`) 以点亮百科。
   - 同步至 `Game1.stats.ItemsShipped`（全物品发货成就）。

### 阶段四：清晨更迭与唤醒 (Dawn Progression)
玩家确认了所有 UI 后，最后一步黑屏执行环境系统的全局刷新：
- 时间与日期步进，幸运值重置。
- 机器生产推进、建筑进度推进。
- 作物判定水源并生长，杂草、浆果、树木刷新。
- 播放随机夜晚事件（陨石坠落、女巫、仙子）。
- 解除维度冻结，时间调回 6:00 AM，重置玩家行动状态，宣告白天开始。

## 3. 下一步执行计划（我们先从哪里开始）

考虑到整个流程十分庞大，我们将拆解执行。
**首要目标（Phase 1）：构建“夜间结算系统” (Night Settlement & UI Stack)**
1. **网络通信基建**：建立一个通用的 `OvernightSettlementPayload`，将服务端的技能提升数据和今日出货清单打包发送给客户端。
2. **构建复合式界面管理器**：由于我们有 `升级界面` 和 `出货界面`，需要一个管理器在关闭上一个 GUI 时自动拉开下一个（模拟原版栈机制）。
3. **完成配方/图鉴/状态的瞬间结算**：当玩家在这些前端界面点击“OK”时，通过发包告知服务端确立职业、增减血量、以及将统计数据计入图鉴系统。

准备就绪后，随时可以切入第一步的代码编写。
