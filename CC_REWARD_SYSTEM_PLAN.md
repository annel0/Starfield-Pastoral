## 各系统详细规划

### 1. 温室修复 (Pantry → Greenhouse)

**SDV 效果：** 农场上的废弃温室变为可用建筑，内部全年无季节限制，可种植任何作物。

**MC 落实思路：**
- 温室是一个独立的室内子空间（类似现有的 NPC 房屋）
- 完成前：温室外观为废墟状态（破损屋顶、杂草），门不可交互
- 完成后：执行一次"翻新"替换（复用现有 CC 的 ruins→refurbished 逻辑），温室变为可进入
- 温室内部：约 10×12 的种植区域，使用特殊方块或标记使其忽略季节检查
- **关键决策：** 温室内的作物系统需要跳过季节枯萎逻辑。可以用区域标记（类似矿洞维度的特殊处理）让温室内的 `StardewCropBlock` 的 `seasonCheck()` 始终返回 true

**依赖：** 
- 温室建筑模型/schematic
- 农场区域上的温室占位方块
- 季节豁免机制

---

### 2. 采石场桥梁修复 (Crafts Room → Quarry Bridge)

**SDV 效果：** 山区通往采石场的断桥修复，玩家可以通过桥到达采石场区域采集矿石和木材。

**MC 落实思路：**
- 在山区区域预留一处断桥位置（用半断的木板/栏杆表达"不可通过"）
- 完成后：桥梁方块替换为完整的木桥，移除隐形屏障或不可通过方块
- 桥另一侧是采石场区域 — 一片露天矿石刷新区（定期生成石头/矿石/宝石节点）
- **简化方案：** 如果采石场本身还没做，可以先只做"桥修好了"的视觉变化 + 道路打通，采石场内容后续填充

**依赖：**
- 山区地图上的断桥/完整桥 schematic
- 采石场区域设计（可后做）
- 屏障方块系统

---

### 3. 闪光巨石移除 + 淘金 (Fish Tank → Glittering Boulder / Panning)

**SDV 效果：** 矿洞入口旁的发光巨石被移除，河流中开始出现闪光亮点，玩家用铜锅淘金可获得矿石/宝石。

**MC 落实思路：**
- **巨石移除：** 在山区河流旁放置一个特殊巨石方块（发光材质），完成后删除该方块
- **淘金系统：** 
  - 河流中定时随机生成"闪光水面"方块/粒子标记
  - 玩家手持铜锅右键闪光点 → 打开淘金战利品表（矿石、宝石、混合种子等）
  - 淘金工具需要单独注册（铜锅 = 新 Item）
- **分阶段实施：** 先做巨石移除的视觉效果，淘金玩法可以作为独立 feature 后续接入

**依赖：**
- 闪光巨石方块 + 模型
- 铜锅物品
- 淘金战利品表
- 河流闪光点生成逻辑

---

### 4. 矿车系统修复 (Boiler Room → Minecarts)

**SDV 效果：** 4 个矿车站点恢复运行（矿洞 ↔ 小镇 ↔ 采石场 ↔ 农场公交站），实现快速传送。

**MC 落实思路：**
- 4 个矿车方块分布在指定位置：
  - 矿洞入口附近
  - 小镇（铁匠铺旁）
  - 采石场（桥修好后才有意义）
  - 农场巴士站附近
- 完成前：矿车方块存在但右键提示"矿车已损坏"
- 完成后：右键打开传送选择 GUI，选择目的地后传送玩家到对应矿车位置
- **实现：** 本质是一个带 GUI 的传送系统。矿车方块 → 打开菜单 → 选目的地 → `player.teleportTo()`
- **注意：** 矿车之间不需要铁轨相连，SDV 里也是瞬时传送

**依赖：**
- 矿车方块 + 模型（GeckoLib 或静态方块）
- 传送选择 GUI
- 4 个站点坐标配置

### 5. 沙漠巴士修复 (Vault → Desert Bus)

**MC 落实思路：**

- 完成前：巴士外观破旧，不可交互

- 沙漠是一个全新的大区域，包含：骷髅洞穴、沙漠商人 Sandy、椰子树等
- **建议留到最后：** 沙漠本身内容量巨大，巴士只是入口。可以先做巴士修复的视觉变化，沙漠区域作为独立大版本

**依赖：**
- 巴士方块模型（损坏/完好两套）
- Pam NPC 驾驶位调度
- 沙漠区域（大工程，可后做）

## 通用基础设施

所有区域奖励共用的底层组件：

### 邮件标记系统
- 已有 `PlayerStardewData.mailFlags` 存储 `ccPantry` 等标记  
- 需要：区域完成时自动写入标记 → 次日触发对应效果
- **AreaRewardDispatcher** — 一个统一入口，监听区域完成事件，根据 areaId 分发奖励逻辑

### 世界状态切换
- 多个奖励涉及"完成前/完成后"的方块替换（温室、桥、巨石、巴士、矿车）
- 可复用现有的 `AreaRestoreHandler` 逻辑：用 schematic 定义两套状态，完成后执行替换
- 需要：`WorldRewardApplier` — 在玩家加入世界时检查所有邮件标记，确保世界状态与进度一致

### 持久化 & 同步
- 标记存在 `CommunityCenterSavedData` 或 `PlayerStardewData` 中
- 世界加载时需要重新应用已解锁的效果（温室可进入、矿车可用、桥存在等）

## 推荐实施顺序

```
Phase 1 — 快速见效（纯数据 + 简单方块替换）
  ├── 公告板友好度奖励（纯数据，最快）
  ├── 闪光巨石移除（删一个方块）
  └── 奖励分发框架 (AreaRewardDispatcher)

Phase 2 — 传送类系统
  ├── 矿车系统（4 个传送点 + GUI）
  └── 采石场桥梁修复（方块替换 + 屏障移除）

Phase 3 — 建筑类系统
  └── 温室修复（子空间 + 季节豁免 + 种植区）

Phase 4 — 远期（依赖大量新内容）
  ├── 沙漠巴士 + 沙漠区域
  └── 淘金玩法（铜锅 + 河流闪光点）
```

## 技术风险 & 注意事项

1. **世界状态一致性：** 玩家重新加载世界时，所有已完成的奖励效果必须重新应用。不能只在"完成那一刻"执行一次方块替换 — 需要持久化标记 + 加载时检查
2. **多人游戏：** 邮件标记是 per-player 还是全局？SDV 是主玩家的标记控制世界状态。建议跟随现有 `CommunityCenterSavedData`（全局）
3. **温室季节豁免：** 需要确保作物系统有"区域级别"的季节覆盖能力，而不是硬编码维度检查
4. **矿车 GUI：** 需要新的 MenuType 注册，但逻辑很简单（4 个按钮 → 4 个坐标）
5. **巴士最后做：** 沙漠是整个游戏后期的核心区域，不要因为"巴士奖励"就急着做沙漠内容
