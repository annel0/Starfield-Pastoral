# 全体 NPC 日程原版一致性总规划

## 0. 目标
构建一套尽可能复现 Stardew 原版行为的日程系统，使用 wiki 辅助理解，并保证运行结果可追踪、可解释、可复现。

硬性约束：
- 不允许凭空编造地点映射。
- 不允许隐藏锚点猜测。
- 不允许使用生物 AI 寻路来执行日程移动。
- 所有日程决策必须能被调试日志完整解释。

## 1. 原版等价规则（权威）

### 1.1 日程 key 选择顺序（普通 NPC）
严格按“首个命中”顺序：
1. festivalId_day 和 festivalId
2. season_day
3. day_hearts（同日多档心值取最高）
4. day
5. bus（仅 Pam 特例）
6. rain2
7. rain
8. season_weekday_hearts（同日多档心值取最高）
9. season_weekday
10. weekday_hearts（同日多档心值取最高）
11. weekday
12. season
13. spring_weekday
14. spring
15. default

规则说明：
- key 匹配不区分大小写。
- Sun 只表示星期日，不表示天气“晴天”。
- 若没有可用 key：回退到 spring，再回退 default。

### 1.2 脚本命令语义
- GOTO key：递归跳转，必须有循环保护。
- NOT friendship：按好感阈值控制分支可用性。
- MAIL 条件：按信件/旗标状态控制分支。
- bed 位置：按“回家规则”解析最终落点。
- 时间 token 使用 HHMM 语义。

### 1.3 节点语义
每个节点包含：
- checkpoint 时间
- location id
- tile x/y
- 朝向 facing
- 可选动画 token
- 可选对话 token

## 2. 数据架构（禁止猜测）

### 2.1 三层模型
第 A 层：原版日程源
- 每个 NPC 的导入 schedule JSON。
- 保留原始 key 和原始节点 token，不改写语义。

第 B 层：语义归一化
- 解析为标准化日程图：
  - keys
  - conditions
  - goto 边
  - 有序节点列表

第 C 层：世界绑定
- 由项目 owner 与评审共同维护显式绑定文件。
- 绑定负责把 schedule location/tile 意图映射到世界中的具体点位。

### 2.2 绑定格式
按“日程地点 + 用途”定义：
- scheduleLocation（如 CommunityCenter）
- endpointId（团队显式指定）
- endpointType（interiorDoor/hallSpot/counterSpot/plazaSpot）
- dimension
- x y z
- radius

若 endpoint 缺失：NPC 不移动，并输出硬警告日志。

## 3. 路线系统（自定义寻路）

### 3.1 寻路要求
- 用自定义路径运行时替代实体导航。
- 在可通行网格上执行 A*。
- 以下情况必须重算路径：
  - 目标改变
  - 障碍出现
  - 卡住超时
- 由 waypoint 跟随器驱动移动。

### 3.2 路线策略
- 起点：NPC 当前坐标或最近已声明 endpoint。
- 终点：当前日程节点绑定的 endpoint。
- 中间路径：由 A* 自动生成。

这满足：
- 起点/终点由你控制。
- 中间路径交给系统自动计算。

## 4. wiki 辅助工作流（抄什么、怎么抄）

### 4.1 wiki 的用途
- 理解分支语义和触发条件。
- 校验天数/季节行为是否符合预期。
- 交叉检查可疑 key 与特殊日期行为。

### 4.2 必须从游戏数据抄的内容
- 精确 key 和节点行必须来自导入的原版日程数据，禁止手写重编。

### 4.3 冲突处理策略
若 wiki 描述与原始 schedule 行冲突：
- 以原始 schedule 行为准。
- 把 wiki 差异记录到审计日志备注。

## 5. 全局实施阶段

阶段 1：确定性解析器
- 重建严格顺序的 key 解析器。
- 增加 key 候选链路 trace。
- 增加 goto/condition trace。

阶段 2：绑定闸门
- 全局禁用锚点推断回退。
- 运行时必须要求显式 endpoint 绑定。

阶段 3：自定义路径引擎
- 接入 A* 规划器和 waypoint 跟随器。
- 移除对 Mob 导航的依赖。

阶段 4：Lewis 试点
- 单 NPC 端到端闭环，开启严格诊断。
- 通过一致性检查后再扩展。

阶段 5：批量推广
- 按 NPC 分批扩展。
- 每一批通过 parity checklist 后进入下一批。

## 6. 必要调试可观测性
每个 NPC 调试输出必须包含：
- 当前 day/season/weather
- key 候选顺序
- 命中的 key 与命中原因
- 被拒绝 key 与拒绝原因
- 当前节点 index 与 checkpoint
- 绑定 endpoint id 与世界坐标
- 当前路径长度与下一 waypoint
- 卡住检测状态与最近一次重算 tick

## 7. 你需要做什么（Owner 任务）
1. 批准严格一致性策略：
- 原始 schedule 行是唯一事实源。
- 禁止隐式锚点回退。

2. 定义 endpoint 目录（按区域先后）：
- 先做高频室内/室外目的地。
- 每个有意义目的地至少一个 endpoint id。

3. 周期性审核一致性报告：
- 对比“期望路线”和“运行实测路线”的不一致清单。

4. 决策特殊日策略：
- 度假村日
- 诊所日
- 节日日
- 地图不可用时的替代目的地

## 8. 我需要做什么（Agent 任务）
1. 实现严格解析器与链路追踪。
2. 增加“必须绑定 endpoint”运行时闸门。
3. 实现自定义 A* 路径服务。
4. 实现日程一致性审计命令。
5. 完成 Lewis 全流程试点并输出不一致报告。
6. 按批次扩展到全部 NPC。

## 9. 完成定义（原版一致）
对每个 NPC/日期场景：
- 选中的 key 与原版语义一致。
- checkpoint 序列一致。
- 能在容差范围内到达目的地。
- 无重复 NPC。
- 正常日程执行期间无 NPC 丢失。
- 调试 trace 能解释全部决策。

## 10. 立即执行冲刺
1. 锁定解析顺序并加入决策 trace。
2. 增加 endpoint 绑定 schema 和校验。
3. 完成 Lewis 端到端（自定义寻路版本）。
4. 产出第一版一致性证据包（日志 + 时间线）。

## 11. 下一步冲刺（对话 + 送礼 + 好感）
目标：从“可显示文本”升级到“原版代码语义一致”。

### 11.1 方法级落地清单（必须按序）
1. 对齐 `NPC.loadCurrentDialogue`：
- 建立与原版同优先级的对话选择链（天气、季节、星期、节日、好感、已读状态、随机分支）。
- 保证同一天内对话队列消费行为与原版一致（首句、后续句、重复交互回退）。

2. 对齐 `NPC.getGiftTasteForThisItem`：
- 完成原版 taste 分类与例外规则（love/like/neutral/dislike/hate）。
- 保留当前项目兼容过滤，但过滤只在加载阶段做，不改变已支持条目的原版判定逻辑。

3. 对齐 `NPC.receiveGift`：
- 实现每周赠礼上限、每日赠礼状态、生日倍率、事件互斥与禁止赠礼场景。
- 同步礼物后即时反馈（文本、音效、表情触发点）与原版流程顺序。

4. 对齐 `NPC.grantConversationFriendship`：
- 实现每日首次对话加好感与门控规则。
- 保证与赠礼系统在同日叠加时的顺序一致。

### 11.2 验收断言（最小闭环）
1. 同一 NPC、同一天、同一时间点，重复对话 3 次：分支与回退行为可复现。
2. 一周内连续赠礼：第 3 次被拒绝或不加分行为与原版一致。
3. 生日当天送同一礼物：好感增量与平日倍率一致。
4. 日切后首次对话与赠礼状态重置正确。

### 11.3 可观测性（新增强制日志）
1. 对话决策 trace：候选键、命中键、拒绝原因。
2. 赠礼判定 trace：taste 分类、倍率、是否计入周上限。
3. 好感变更 trace：来源（对话/送礼）、delta、变更前后数值。
